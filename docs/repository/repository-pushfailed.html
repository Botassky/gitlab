プッシュに失敗する原因と対応¶

プッシュに失敗したときの原因と対応方法です。

● 原因と対応
● 失敗と対応の例 #1 ： 同一ファイルの更新（コンフリクトあり）
　　・ ファイルを編集
　　・ コミット
　　・ プッシュ　→　エラー発生
　　・ プル　→　コンフリクト発生
　　・ コンフリクトが発生したファイルの修正
　　・ 再コミット
　　・ 再プッシュ
　　・ リモートリポジトリの確認
● 失敗と対応の例 #2 ： 同一ファイルの更新（コンフリクトなし）
　　・ ファイルを編集
　　・ コミット
　　・ プッシュ　→　エラー発生
　　・ プル
　　・ 取り込んだファイルの確認
　　・ 再コミット
　　・ 再プッシュ
　　・ リモートリポジトリの確認
● 失敗と対応の例 #3 ： 異なるファイルの更新
　　・ ファイルを編集
　　・ コミット
　　・ プッシュ　→　エラー発生
　　・ プル
　　・ 取り込んだファイルの確認
　　・ 再コミット
　　・ 再プッシュ

原因と対応¶

原因

  • リモートブランチよりもリモート追跡ブランチの履歴が古いため。

  • 「履歴が古い　＝　ファイルが古い」状態なので、そのままプッシュするとリモートリポジトリ内のファイルがデグレードする可能性があります。そのため、リモートブランチよりリモート追跡ブランチの履歴が古い場合にプッシュは失敗します（ファイルの上書きによるデグレードを防止します）。

対応

  • プルを実行し、リモートリポジトリの変更内容をローカルリポジトリに反映します。

  • プルの実行時にコンフリクトが発生したときは、優先する変更内容を決定します。その後「コミット　→　プッシュ」を実行し、リモートリポジトリに変更内容を反映します。

  • プルの実行時にコンフリクトが発生しなかったときは、Git がコミット内容に矛盾が出ないようファイルをマージします。その後「プッシュ」を実行し、リモートリポジトリに変更内容を反映します。

失敗と対応の例 #1 ： 同一ファイルの更新（コンフリクトあり）¶

ファイルを編集¶

index.rst ファイル内の「発足」を変更します。

↓

コミット¶

編集内容をコミットします。

  1. コミット前の状態　→　コミットされていない変更があります をクリック


  2. 対象ファイルをステージングエリアに登録　→　コミット をクリック


  3. コメントを入力　→　コミット をクリック


  4. コミット終了


プッシュ　→　エラー発生¶

ローカルリポジトリの内容をリモートリポジトリに反映します。今回はエラーが発生し、プッシュに失敗します。

  1. プッシュ をクリック


  2. ローカル と リモートブランチ の値を確認　→　プッシュ をクリック


  3. プッシュ中にエラー発生　→　閉じる をクリック


    ヒントと翻訳結果です。

    Updates were rejected because the remote contains work that you do not have locally. This is usually caused by another repository pushing to the same ref. You may want to first integrate the remote changes (e.g., ‘git pull …’) before pushing again. See the ‘Note about fast-forwards’ in ‘git push –help’ for details.

    ↓

    リモートにローカルにない作業が含まれているため、更新が拒否されました。これは通常、他のリポジトリが同じ ref にプッシュしていることが原因です。最初にリモートの変更を統合してから (たとえば ‘git pull ….’ のように) 再度プッシュしたほうがいいかもしれません。詳細は ‘git push –help’ の ‘fast-forward についての注意事項’ を参照ください。


  4. キャンセル をクリック


  5. プッシュ終了

    プッシュに失敗したので、リモートリポジトリの内容は変更されていません。リモート追跡ブランチも更新されないため、プッシュ前と同じ状態です。


プル　→　コンフリクト発生¶

リモートリポジトリの変更をローカルリポジトリに反映します。今回はコンフリクトが発生します。

コンフリクト（衝突）

同じファイルの同じ箇所の修正がかち合った（コンフリクト、衝突）状態のこと。

  1. リモートリポジトリの状態を確認

    前回の プル の後に、新しいプッシュがあることを確認できます。上記のエラーは履歴がリモート追跡リポジトリよりリモートリポジトリが新しかったために発生したことがわかります。


  2. プル をクリック


  3. プル元を確認　→　プル をクリック


  4. プル中にコンフリクトが発生　→　閉じる(C) をクリック


  5. プル終了

    プル中にコンフリクトが発生しましたが、ローカルリポジトリにリモートリポジトリの変更内容が取り込まれました。 ！ マークが付いているのがコンフリクトが発生したファイルです。


コンフリクトが発生したファイルの修正¶

  1. コンフリクトが発生した docs/source/index.rst ファイルを開く

    コンフリクトマーカー（衝突マーカ－）の確認　＝　コンフリクトが発生した部分が対象ごとに色分けして編みかけされています。


  2. 対応方法の候補から適切なものを選択

    次の 3 つの対応方法の候補が表示されています。
    ・ 現在の変更を取り込む　＝　ローカルリポジトリの内容を取り込みます
    ・ 入力側の変更を取り込む　＝　リモートリポジトリの内容を取り込みます
    ・ 両方の変更を取り込む　＝　両リポジトリの内容を取り込みます
    今回は「入力側の変更を取り込む」を選択します。

  3. 修正内容の保存

    選択した候補の内容だけが表示されています。他に修正が必要な箇所があれば、このタイミングで修正します。修正後はファイルをセーブします。


再コミット¶

  1. 修正したファイルをステージングエリアに移動


  2. コミット をクリック


  3. コミット をクリック

    コミット時のコメントは自動的に設定されます。変更も可能です。


  4. コミット終了


  5. master が先頭の履歴を指していることを確認


再プッシュ¶

  1. プッシュ をクリック


  2. ローカルブランチ と リモートブランチ の値を確認　→　プッシュ をクリック


  3. プッシュ終了

    ローカルリポジトリとリモートリポジトリが同じ内容になったので、 master ・ origin/master ・ origin/HEAD が同じ位置に表示されています。


リモートリポジトリの確認¶

コミット履歴に今回プッシュした内容が追加されたことを確認できます。

プル¶

リモートリポジトリの変更をローカルリポジトリに反映します。

  1. リモートリポジトリの状態を確認

    自分（ riki ）が行ったプッシュの後に新しいプッシュがあることを確認できます。上記のエラーは履歴がリモート追跡リポジトリよりリモートリポジトリが新しかったために発生したことがわかります。


  2. プル をクリック


  3. プル元を確認　→　プル をクリック


  4. プル終了

    ローカルリポジトリにリモートリポジトリの変更内容が取り込まれました。 ！ マークが付いているファイルがないので、コンフリクトは発生していません。


取り込んだファイルの確認¶

  • コンフリクトが発生していないので、色分けされた部分はありません。

  • リモートリポジトリ（青枠で囲んだ部分）とローカルリポジトリ（赤枠で囲んだ部分）の変更箇所がマージされました。


再コミット¶

コミットされていない変更があります が表示されていないので、再コミットは不要です。


再プッシュ¶

  1. プッシュ をクリック


  2. ローカルブランチ と リモートブランチ の値を確認　→　プッシュ をクリック


  3. プッシュ終了

    ローカルリポジトリとリモートリポジトリが同じ内容になったので、 master ・ origin/master ・ origin/HEAD が同じ位置に表示されています。

  2. 対象ファイルをステージングエリアに登録　→　コミット をクリック


  3. コメントを入力　→　コミット をクリック


  4. コミット終了


プッシュ　→　エラー発生¶

ローカルリポジトリの内容をリモートリポジトリに反映します。今回はエラーが発生し、プッシュに失敗します。

  3. プッシュ中にエラー発生　→　閉じる をクリック


    ヒントと翻訳結果です。

    Updates were rejected because the tip of your current branch is behind its remote counterpart. Integrate the remote changes (e.g. ‘git pull …’) before pussing again. See the ‘Note about fast-forwards’ in ‘git push –help’ for details.

    ↓

    現在のブランチの先端がリモートブランチの後ろにあるため、更新が拒否されました。リモートの変更を統合してから (たとえば ‘git pull ….’ のように) 再度 pussing しましょう。詳細は ‘git push –help’ の ‘fast-forward についての注意事項’ を参照ください。


  4. キャンセル をクリック


  5. プッシュ終了
